# 学籍管理系统课程设计报告

## 1. 课题背景

随着高校教学管理信息化水平不断提升，传统依赖纸质材料和分散表格的学籍管理方式已经难以满足准确性、时效性和规范性的要求。学生档案、课程安排、选课记录、成绩管理、奖惩记录和学籍异动等数据量较大，业务之间关联紧密，若缺乏统一的信息系统支持，容易出现数据重复、信息不一致和统计困难等问题。

本课程设计围绕“学籍管理系统”这一典型数据库应用场景，设计并实现一个基于 `C/S` 架构的桌面端系统。系统采用 `Python + PyQt5 + MySQL` 技术路线，通过图形界面完成用户操作，通过关系数据库完成业务数据管理，体现数据库原理课程中需求分析、E-R 建模、关系模式设计、完整性约束、数据库编程以及应用开发等核心知识。

## 2. 设计目标

本系统的总体目标是实现一个能够实际运行和演示的学籍管理系统，并满足课程设计对数据库应用开发的完整要求。具体目标如下：

- 采用 `C/S` 架构实现一个桌面端数据库应用系统；
- 使用 `MySQL` 作为后台数据库；
- 完成需求分析、E-R 图设计、逻辑结构设计和系统实现；
- 支持管理员与学生两类用户；
- 覆盖学生档案、课程、选课、成绩、奖惩、学籍异动和附件管理等核心业务；
- 将图片、视频、普通文件纳入系统管理范围；
- 合理使用存储过程、函数、事务和触发器四类数据库高级特性；
- 形成可提交的课程设计报告、数据库脚本、源代码和测试材料。

## 3. 需求分析

### 3.1 用户角色分析

系统包含两类用户。

1. 管理员  
管理员负责系统中的主要业务维护工作，包括学生基本信息管理、课程管理、选课管理、成绩录入与修改、奖惩信息管理、学籍异动登记以及学生附件上传与删除等。

2. 学生  
学生登录后只具备查询权限，可查看个人基本信息、课程与成绩、奖惩记录、学籍异动记录以及个人附件信息。

### 3.2 功能需求分析

系统主要功能需求如下：

1. 登录功能  
系统提供统一登录界面，支持用户选择“管理员”或“学生”角色后登录。系统根据不同角色验证对应数据表中的账号信息，并进入不同主界面。

2. 学生档案管理  
管理员可对学生信息进行新增、修改、删除和查询。学生信息包括学号、姓名、性别、联系方式、邮箱、入学年份、学籍状态、所属专业、所属班级等。

3. 专业与班级信息管理支撑  
系统采用专业表和班级表作为基础数据支撑，学生档案与其建立外键关联。这样既能够减少数据冗余，又便于后续扩展统计分析功能。

4. 课程管理  
管理员可维护课程编号、课程名称、学分、学时和开课学期等信息。

5. 选课管理  
管理员可为学生增加选课记录。系统需防止同一学生对同一课程重复选课。

6. 成绩管理  
管理员可在已有选课记录基础上录入或修改成绩，系统自动计算总评并给出等级结果。学生登录后可以查看本人课程及成绩信息。

7. 奖惩管理  
管理员可登记奖励或惩罚记录，学生可查看自身奖惩信息。

8. 学籍异动管理  
管理员可对学生进行专业变更、班级变更、状态变更等操作。系统需要保证异动记录与学生主档信息同步更新。

9. 附件管理  
管理员可为学生上传图片、视频和普通文件，系统需保存文件元数据并维护物理文件路径。学生可查看本人附件记录。

10. 概览统计  
管理员端主界面显示学生总数、课程总数、选课记录数和附件数，用于快速展示系统基本运行情况。

### 3.3 非功能需求分析

- 界面简洁，便于实验课展示；
- 业务流程清晰，适合教师快速验收；
- 数据具有完整性和一致性约束；
- 数据库设计满足第三范式要求；
- 可在 Windows 本地环境中独立部署运行；
- 源码结构清晰，便于说明与维护。

## 4. 可行性分析

### 4.1 技术可行性

系统所涉及业务属于典型的管理信息系统，数据实体清晰，关系明确，非常适合采用关系数据库进行建模和实现，因此技术路线具备较高可行性。

## 5. 总体设计

### 5.1 系统架构设计

本系统采用 `C/S` 架构。

- `Client`：使用 `PyQt5` 编写桌面客户端，负责界面显示、用户输入和交互逻辑；
- `Server`：本设计不单独开发应用服务器，直接以本机 `MySQL` 数据库作为数据服务端；
- 客户端通过数据库访问层与 `MySQL` 通信，实现数据的读取和更新。

系统总体架构如下：

1. 用户在客户端输入数据；
2. 界面层调用业务服务层；
3. 业务服务层调用 DAO 层；
4. DAO 层执行 SQL 访问 `MySQL`；
5. 数据结果返回给客户端界面显示。

### 5.2 模块划分

系统划分为以下功能模块：

- 登录模块
- 系统概览模块
- 学生管理模块
- 课程管理模块
- 选课与成绩模块
- 奖惩管理模块
- 学籍异动模块
- 附件管理模块
- 学生查询模块

### 5.3 代码分层设计

为提高系统结构清晰度，项目采用分层设计：

- `app/ui`：界面层，负责窗口、表格、按钮和交互事件；
- `app/services`：业务服务层，负责业务流程封装和附件文件处理；
- `app/dao`：数据访问层，封装全部 SQL 操作；
- `app/db`：数据库连接层，管理连接、查询、事务；
- `app/config`：配置层，负责读取数据库连接与存储路径；
- `app/utils`：工具层，负责成绩计算和附件类型识别等通用逻辑。

## 6. 数据库设计

### 6.1 E-R 设计说明

系统以学生实体为核心，围绕专业、班级、课程、选课、成绩、奖惩记录、学籍异动记录和附件记录构建完整数据模型。  
主要实体关系如下：

- 一个专业包含多个班级；
- 一个专业可对应多个学生；
- 一个班级可对应多个学生；
- 学生与课程之间是多对多关系，通过选课表建立联系；
- 每条选课记录对应一条成绩记录；
- 学生与奖惩记录、异动记录、附件记录均为一对多关系。

E-R 图可见 [diagrams.md](E:\database\student_management_system\docs\diagrams.md)。

### 6.2 数据表设计

系统主要数据表如下：

1. `major`  
保存专业信息。

2. `class`  
保存班级信息，并与专业关联。

3. `admin_user`  
保存管理员登录账号与密码散列值。

4. `student`  
保存学生基础档案信息，是系统核心表之一。

5. `course`  
保存课程基础信息。

6. `course_selection`  
保存学生选课信息，用于连接学生与课程。

7. `score`  
保存成绩数据，包括平时分、期末分、总评和等级。

8. `score_audit_log`  
保存成绩变更日志，用于体现触发器功能。

9. `reward_punishment`  
保存学生奖惩信息。

10. `student_status_change`  
保存学生学籍异动记录。

11. `student_attachment`  
保存学生附件元数据，包括文件名、文件类型、路径和说明。

### 6.3 规范化分析

本系统数据库模式满足第三范式，原因如下：

- 所有表均设置了主键；
- 表中所有非主属性都完全依赖于主键；
- 不同业务内容被拆分到独立数据表中，避免了传递依赖；
- 专业、班级、课程等基础数据单独维护，减少了冗余和更新异常。

例如，如果把课程名称直接存储在成绩表中，会导致课程名称重复保存并带来更新异常；通过将课程独立成表并以外键关联，能够避免这一问题。

### 6.4 完整性约束与索引设计

系统使用了以下完整性机制：

- 主键约束：保证每条记录唯一；
- 外键约束：保证学生、课程、选课、成绩、奖惩、异动、附件之间关系正确；
- 唯一约束：`student.student_no`、`course.course_code`、`admin_user.username` 等字段不可重复；
- 联合唯一约束：`course_selection(student_id, course_id)` 防止重复选课；
- 枚举约束：对性别、学籍状态、奖惩类型、附件类型、异动类型等进行值域限制；
- 索引：在姓名、专业、课程号及关联字段上建立索引，提升查询性能。

## 7. 数据库高级特性设计

### 7.1 存储过程设计

系统设计了存储过程 `sp_upsert_student_profile`，用于根据学号判断学生信息是新增还是更新。

设计目的：

- 避免在应用层分别编写大量新增和更新 SQL；
- 保证学生档案写入逻辑统一；
- 体现数据库层对复杂写入流程的封装能力。

应用场景：

- 管理员新增学生；
- 管理员修改学生信息。

### 7.2 函数设计

系统设计了两个函数：

1. `fn_calculate_total_score`  
根据平时分与期末分按 40% 和 60% 权重计算总评。

2. `fn_score_to_level`  
根据总评分数返回“优秀、良好、中等、及格、不及格”五级等级结果。

设计意义：

- 保证成绩计算逻辑统一；
- 避免应用层和数据库层出现多套评分规则；
- 使成绩录入更符合数据库编程要求。

### 7.3 事务设计

系统在“学籍异动登记”功能中使用事务。  
在一次异动操作中，需要同时完成两件事：

1. 向 `student_status_change` 表插入异动记录；
2. 更新 `student` 表中的专业、班级或学籍状态。

若只完成其中一步，会导致数据不一致。通过事务处理，可确保：

- 所有步骤都成功时统一提交；
- 任意一步失败时整体回滚。

### 7.4 触发器设计

系统设计了两个成绩相关触发器：

1. `trg_score_before_update`  
在成绩更新前自动修改 `last_updated` 字段，记录最近更新时间。

2. `trg_score_after_update`  
在成绩更新后自动向 `score_audit_log` 表插入一条日志记录，保存旧总评和新总评。

设计意义：

- 自动记录成绩变更过程；
- 体现数据库层的自动审计能力；
- 便于答辩时展示触发器实际效果。

## 8. 系统实现说明

### 8.1 登录模块实现

登录窗口由 [login_window.py](E:\database\student_management_system\app\ui\login_window.py) 实现。用户选择角色并输入账号密码后，系统调用 `AuthService` 完成认证。认证逻辑会对输入密码进行 `SHA-256` 散列，再与数据库中的散列值比较。

管理员认证数据来自 `admin_user` 表，学生认证数据来自 `student` 表。

### 8.2 管理员端实现

管理员主窗口由 [admin_window.py](E:\database\student_management_system\app\ui\admin_window.py) 实现，采用标签页组织各功能模块：

- 概览
- 学生管理
- 课程管理
- 选课与成绩
- 奖惩管理
- 学籍异动
- 附件管理

管理员可在对应标签页中通过表单完成录入和修改操作。系统中常用的数据录入弹窗由 [dialogs.py](E:\database\student_management_system\app\ui\dialogs.py) 统一封装。

### 8.3 学生端实现

学生端主窗口由 [student_window.py](E:\database\student_management_system\app\ui\student_window.py) 实现，主要提供只读查询功能。  
学生登录后可以查看：

- 个人信息
- 课程与成绩
- 奖惩记录
- 学籍异动
- 个人附件

### 8.4 数据访问实现

数据访问层由 [system_dao.py](E:\database\student_management_system\app\dao\system_dao.py) 实现。该文件统一管理所有核心 SQL，包括：

- 登录查询
- 学生增删改查
- 课程增删改查
- 选课记录查询和新增
- 成绩录入和修改
- 奖惩记录维护
- 学籍异动维护
- 附件元数据维护
- 学生端查询

### 8.5 数据库连接实现

[connection.py](E:\database\student_management_system\app\db\connection.py) 封装了：

- 数据库建立连接
- 自动重连
- 查询单条和多条数据
- 执行写操作
- 执行事务

这样的设计使得上层业务逻辑不必直接处理连接和提交细节。

### 8.6 附件管理实现

附件业务由 [system_service.py](E:\database\student_management_system\app\services\system_service.py) 负责。

实现方式如下：

1. 管理员选择本地文件；
2. 系统根据文件后缀识别为图片、视频或普通文件；
3. 系统将文件复制到 `attachments/{student_id}/` 目录；
4. 将文件名、类型、路径、上传时间和说明写入 `student_attachment` 表。

这种方案实现简单，满足课程要求，也避免了把大文件直接存入数据库带来的复杂度和性能问题。
